# #031 Realtime の差分更新への移行（スケーラビリティ改善）

## 概要

現在は Realtime イベントを受け取るたびに `fetchMatches()` で全件を再取得している。
1大会あたりの試合数と同時接続人数が増えると、コストが `試合数 × 同時接続人数` で増加する。

スケールが見込まれる場合は、`postgres_changes` の `payload` を使った差分更新に移行する。

**現時点では対応不要**。1大会あたり数十局・同時接続数人の規模では全件再取得で問題ない。
スケールアップが必要になったタイミングで着手する。

## 現状のトレードオフ

| | 全件再取得（現在） | 差分更新（payload活用） |
|---|---|---|
| 実装のシンプルさ | ◎ | △ |
| データ量が増えたときの速度 | △ | ◎ |
| 正確性 | ◎ | △（JOIN結果が payload に含まれないため補完が必要） |

## 差分更新が難しい理由

`postgres_changes` の `payload.new` にはそのテーブルの1行のみが入る。
`useMatches` は `tables`・`players` を JOIN して取得しているため、payload だけでは `toMatch()` の変換ができない。

差分更新を実装するには、`payload.new` の `match_id` や `player_id` をキーに、
キャッシュ済みのプレイヤー・卓データと結合して State を手動で組み立てる必要がある。

## やること（着手時に詳細化）

- `matchChannel` / `resultsChannel` のハンドラで `payload.new` を使って State を更新する
- `tables`・`players` のキャッシュを `useMatches` の外（Context 等）に持ち出して参照できるようにする
- State の整合性が崩れたときのフォールバックとして全件再取得を残す
