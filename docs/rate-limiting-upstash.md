# レートリミット: Upstash Redis への移行

作成日: 2026-06-02

---

## 問題: インメモリ実装はサーバーレスで機能しない

元の実装はプロセスのメモリ上の `Map` にリクエスト回数を記録していた。

Vercel のサーバーレス（Functions）はリクエストが増えると複数のインスタンスが同時に起動する。各インスタンスが独立した `Map` を持つため、インスタンスをまたいだカウントの共有ができず、レートリミットが実質機能しない。

```
リクエスト1 → インスタンスA（Map: { ip: 3回 }）
リクエスト2 → インスタンスB（Map: { ip: 3回 }）  ← Aのことを知らない
```

ローカル開発（`next dev`）では1プロセスで動くため問題が出ず、本番で初めて発覚するタイプのバグ。

---

## 解決策: Upstash Redis（`@upstash/ratelimit`）

全インスタンスから参照できる外部ストアにカウントを置く。

### なぜ Supabase ではなく Upstash か

Supabase（既存）も外部ストアとして使えるが不適切。攻撃が集中したとき、Supabase DB へも大量クエリが飛んでしまい、レートリミットが守るべき DB を自分が攻撃する形になる。Redis は専用設計でインメモリのため高速・高耐久。

### リージョンの合わせ方

Vercel の Function リージョン（`hnd1` = AWS ap-northeast-1 Tokyo）に合わせ、Upstash も同じリージョンに配置。リージョンが離れると Upstash との往復が毎リクエスト発生してレイテンシが増える。

---

## 設計上の決断

### フェイルオープンを選択した

Upstash が落ちたとき、例外が伝播すると大会作成・削除が全員 500 になる。

- **フェイルオープン（許可）**: 障害中はレートリミットが無効になるが、アプリは動く
- **フェイルクローズド（拒否）**: 障害中は正常ユーザーも使えなくなる

レートリミットの目的は悪意ある攻撃への対策であり、外部サービスの障害で正常ユーザーを止めるほうが損失が大きいためフェイルオープンを選択。障害検知のため `console.error` でログを残す。

### 遅延初期化が必要

モジュールのトップレベルで `Redis.fromEnv()` を呼ぶと、環境変数が未設定のとき import ごと落ちて `try/catch` が効かない。初回呼び出し時にインスタンスを生成する遅延初期化にすることで、初期化エラーも `try/catch` の対象にする。

```ts
let ratelimit: Ratelimit | null = null;

function getRatelimit(): Ratelimit {
  if (!ratelimit) {
    ratelimit = new Ratelimit({
      redis: Redis.fromEnv(),
      limiter: Ratelimit.fixedWindow(10, "60 s"),
    });
  }
  return ratelimit;
}
```
