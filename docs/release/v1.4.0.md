# v1.4.0 - 大会作成ウィザード化と記録認可モデルの整理

## 麻雀大会スコア v1.4.0

大会作成をセットアップウィザード化して記録開始までを一本道にし、閲覧・記録の URL を分離する記録トークンを一度導入したのち撤去して大会 URL 1本に戻したリリース。あわせてホーム画面のデザインを刷新し、対局保存の待ち時間を短縮、Route Handler のリファクタリングを完了させた。

### 主な変更

#### 大会作成をセットアップウィザード化 (#91)

- 大会作成後に記録を始めるまでの準備（選手登録・ルール設定・記録リンクの共有）が大会ページのボタンに散らばっており、初めて使う人が「次に何をすればいいか」を自力で発見する必要があった
- `/new` を「大会情報 → 選手登録 → ルール選択 → 完了」の4ステップウィザードにし、準備を一本道にした
- 大会レコードは最後にまとめて作成する。players / rules API の認可は「存在する大会のオーナーであること」が前提のため、入力をローカルに保持して一括作成 API に渡すことで途中離脱しても空の大会が残らないようにした
- ルールのプリセットは seed ルールのデフォルトフラグ切り替えで済ませ、カスタム設定時のみ3件目として挿入（デフォルト1件の unique index を維持）
- `/new` ルート追加で大会 ID `new` が到達不能になるため、customId の予約語チェックに `record` / `auth` / `api` もまとめて追加

#### 記録トークンの導入と撤去 (#84, #86, #88, #90 → #92)

- `POST /api/matches` はログイン不要で誰でも書き込めていたため、URL を知っていれば対局結果を偽造できた。これを塞ぐため「主催者が記録リンクを渡した人だけが記録できる」capability として、大会ごとの記録トークンを導入した（サーバー側基盤 #84、`/record` 専用ルートと閲覧ページの読み取り専用化 #86、記録ページへのランキング・履歴タブ追加 #88、PC レイアウト調整 #90）
- トークンは URL の fragment（`#k=`）で受け渡し、`history.replaceState` で即座に消して localStorage に退避。アクセスログや Referer への漏洩を防いだ
- 導入後、実運用で「閲覧はさせたいが記録はさせたくない相手」というケースが一度も発生せず、コストだけが残っていることが分かった。ユーザーが理解すべき概念が3つ（閲覧ページ・記録ページ・記録リンク）に増え、主催者は毎回「どのリンクを誰に渡すか」を考える必要があった
- 大会 ID は推測不能な UUID なので、**大会 URL を知っていること自体を書き込みの capability** として扱う方式に切り替え、トークン基盤・共有モーダル・ページ分離を丸ごと撤去した（-829 行）。カスタム大会 ID も同時に廃止し、URL の推測可能性という前提を崩さないようにした
- 乱用対策はレート制限に一本化。対局の削除・選手/ルール管理は引き続きオーナー専用のまま

#### ホーム画面のデザイン刷新 (#93)

- ベージュ系の配色で全要素の明度が近くカードの輪郭がぼやける、フォント未指定で環境依存の見た目になる、`justify-center` の中央浮きレイアウトがスプラッシュ画面のように見えるといった「間に合わせ」感を解消
- 白ベース + 雀卓グリーンの配色に刷新し Noto Sans JP を導入。CSS 変数の値だけ差し替えたため参照側のコード変更なしに全ページへ反映
- 大会リストを `button` + `router.push` から `Link` に変更し、右クリックで新しいタブを開けるなどリンク本来のセマンティクスを回復

#### パフォーマンス改善 (#94, #95)

- 対局保存時、認可判定に使わない Supabase Auth への問い合わせと、卓・プレイヤー・ルールの確認が直列実行されていたため、外部サービスへのネットワーク往復が待ち時間に積み上がっていた。公開 API では不要なセッション更新をスキップし、3つの DB 確認を並列化
- Upstash Redis のデータベースが消失（ホスト NXDOMAIN）しており、レートリミット呼び出しのたびにデフォルトリトライ（5回・指数バックオフ≒4.3秒）を使い果たしてからフェイルオープンしていたことが判明。復旧ではなく撤去を選択：消失後もレートリミットは実質無効だったが乱用は起きておらず、書き込みの認可（オーナー限定操作はログイン必須、対局記録は推測不能な大会 URL）だけで身内利用には十分と判断
- ローカル実測で `POST /api/matches` が 4.34秒 → 約0.1秒に短縮

#### UI/UX 細部の修正 (#96, #97)

- 点数入力の確定（Enter）キーで東→南→西と次の風の欄へフォーカスを送るようにし、スマホでのキーボード出し直しを削減
- 選手管理・卓登録・`/new` の追加行で、入力欄が縮み切らず追加・保存ボタンが画面外にはみ出す問題を修正。`flex-wrap` でボタンを入力欄の下に折り返す防御的なレイアウトに変更
- `/new` のデスクトップ表示をモバイル前提の `max-w-md` 固定から、`sm` 以上で幅と余白を広げる表示に変更

#### 内部リファクタリング

- Route Handler に残っていた入力検証・認証認可・DB 操作の分離を完了（`AppError` / `route()` による HTTP 層共通基盤 #63、`requireUser` / `requireTournamentOwner` 認可ヘルパー #64、`createMatch` の usecase 化 #70、`create_match_with_results` RPC 化によるトランザクション整合 #74、残りの Route Handler 分離完了 #82）。あわせて migration と生成済み Supabase 型のずれを CI で検出できるようにした
- 環境変数の管理を 1Password (`op run`) に移行し、秘密情報がローカルに平文で残らないようにした (#66)
- 4人ちょうど・単一卓の対局向けにドラッグで席替え＋点数入力する専用 UI を追加。プレイヤー選び直しと回戦番号入力という毎回の手間・入力ミスの余地をなくした (#53)
- `supabase_realtime` publication にテーブルが1つも登録されておらず対局登録がリロードするまで画面に反映されなかった不具合を修正。以後はマイグレーションで管理する (#52)
- 登録・ルール・卓まわりの UX 改善とバグ修正をまとめて実施：一覧更新を Realtime 購読だけに頼らずセルフ refetch を追加、`useOptimistic` による即時反映、並び順の安定化、単一卓大会での卓 UI 非表示など (#50)

### 設計の方針

このリリースの軸は記録トークンの導入と撤去。「閲覧はさせたいが記録はさせたくない相手を守る」という一見もっともな要件のために、fragment 経由のトークン受け渡し・ハッシュ保存・再発行・専用ページという相応の実装を作ったが、実運用にそのケースが一度も現れず、複雑さだけがコストとして残った。大会 ID が推測不能な UUID であるという既存の前提に立ち返り、「URL を知っていること自体を書き込みの capability とする」というより単純なモデルに戻すことで、機能を削って安全性を落とさずにコードを大きく減らせた。過剰な権限分離を先に作ってから、実データを見て要らないと判断し引き返す、という順序そのものがこのリリースの学びになっている。

大会作成のウィザード化も同じ「動線の一本化」という方向性で、準備の分散をなくして記録開始までを一直線にした。パフォーマンス改善は2件とも外部サービス（Auth・Upstash）への不要な同期待ちを削る対応で、特に Upstash は障害時のフェイルオープンが呼び出しごとに数秒のリトライを挟むという設計上の見落としだった。守りたいリスクより実害が大きいと判断した時点で、復旧ではなく撤去を選んでいる。
