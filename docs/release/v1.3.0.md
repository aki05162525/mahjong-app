# v1.3.0 - ルール設定と履歴絞り込み

## 麻雀大会スコア v1.3.0

ハードコードだったウマ・オカを大会ごとのルールとして設定可能にし、対局履歴を名前で絞り込めるようにしたリリース。あわせてサーバーレス環境で機能していなかったレートリミットを修正し、ビルドの安定化と UI の調整を行った。

### 主な変更

#### ウマ・オカを対局ごとに設定可能に (#47)

- これまでウマ `[30,10,-10,-30]`・返し点 25000（オカなし）がハードコードで、卓やリーグごとに異なるルールで打ち分けられなかった
- ルールを大会ごとの第一級エンティティ（`rules`）として持ち、対局時に選択する方式にした。デフォルトを1つ指定でき、対局フォームの初期値になる
- 対局には使ったルール（ウマ・返し点）をスナップショット保存。これにより「対局ごとにルールを変えられる」と「ルールを後から変えても過去対局は再計算しない」を両立
- オカは `(返し点 − 25000) × 4` から導出してトップ群に按分。返し 25000 ならオカ 0 で従来挙動と一致するため、既存大会・対局は移行後も値が変わらない
- 持ち点は4人麻雀の合計 100,000 点固定のため可変にせず、ウマと返し点のみ設定可能

#### 対局履歴を名前で絞り込み (#46)

- 参加者が5人以上になると全員が毎回同卓するわけではなくなり、「自分と特定の人が当たった回」を探すのが大変だった。名前チップによる絞り込みを追加
- 絞り込みは OR ではなく **AND**（選んだ全員が同卓した対局のみ表示）。「自分＋ライバルの直接対決を見たい」というユースケースを表現するための中心的な判断
- AND は「選ぶ人を増やすほど結果が減る」直感に反するため、説明ラベルと0件表示をセットで添えた
- 4人以下は全員が毎回同卓し絞り込みの意味がないため、履歴に登場する重複なしプレイヤーが5人以上のときだけフィルタ UI を表示

#### UI の調整

- 対局入力の座席ラベルを「1人目」などの数字から東南西北の風牌表示に変更（麻雀の座席はそのほうが直感的なため）(#44)
- プラス/マイナスでの色分けをやめ、合計 pt を `--primary` に統一。自動入力された4人目の点数はウォームグレーで「計算済み」を示しつつ主張を抑える。ランキング各統計列は1位の値を強調 (#45)

#### インフラ・バグ修正

- **レートリミットを Upstash Redis に移行 (#42)**: メモリの Map はインスタンスをまたいで共有されず、Vercel のサーバーレス環境ではレートリミットが実質機能していなかった。`@upstash/ratelimit` + `@upstash/redis` に置き換えて全インスタンスでカウントを共有。Upstash 障害時は正常ユーザーを優先してフェイルオープン（`console.error` でログは残す）
- **`supabaseAdmin` を遅延初期化 (#40)**: モジュールトップレベルで `createClient()` を呼ぶと env var 未設定のビルド環境で `pnpm build` がクラッシュしていた。`getSupabaseAdmin()` に変え、リクエスト到達時まで初期化を遅延（Closes #36）
- **Node.js バージョンを `mise.toml` で固定 (#39)**: 開発環境ごとのズレを防止。セキュリティパッチを手動更新なしで取り込めるようメジャーバージョン指定（24）にした
- 依存パッケージの一括更新 (#38)

### 設計の方針

このリリースの中心はウマ・オカのルール化。「対局ごとに変えられる」柔軟性と「過去対局を壊さない」安定性は一見トレードオフだが、ルールを対局にスナップショット保存し計算結果を `match_results` に確定保存することで、対局編集機能の有無に関係なく両立させた。オカを返し点から導出する設計により、既存データはマイグレーション後も無変化を保証している。

履歴の絞り込みは「直接対決を辿る」という具体的なユースケースから AND を選択し、直感に反する挙動を UI のラベルで補った。レートリミットの修正は、サーバーレス環境ではグローバル状態を外部ストアに置かないと機能しないという基盤上の前提を踏まえたもの。
