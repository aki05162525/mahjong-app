# v1.2.1 - Realtime 最適化

## 麻雀大会スコア v1.2.1

Supabase Realtime による対局リアルタイム同期の内部品質を改善したリリース。ユーザー向けの見た目変化はないが、対局登録時に発生する余分なデータベースアクセスを大幅に削減した。

### 主な変更

#### debounce による連続 fetchMatches の抑制 (#29)

- 1試合を登録すると `matches` 1件 + `match_results` 4件の INSERT が走り、Realtime イベントが5連続で届いて `fetchMatches()` が5回呼ばれていた
- 自前の `debounce(100ms)` を導入し、バースト的に届くイベントをまとめて1回の再フェッチに収めた
- アンマウント時は `cancel()` で pending リクエストを破棄してメモリリークを防止

#### match_results に tournament_id を追加して Realtime フィルタリングを有効化 (#30)

- `match_results` テーブルが `tournament_id` を持っていなかったため Supabase Realtime のカラムフィルタが使えず、他大会の対局登録イベントも現在のページに届いていた
- `match_results.tournament_id` カラムを追加し、subscription に `tournament_id=eq.${tournamentId}` フィルタを付与して自大会のイベントのみを受け取るようにした
- `match_id` と `tournament_id` の整合性を composite FK でデータベースレベルから保証（フィルタの正確性の前提条件）

#### Realtime 差分更新への移行で試合登録時の SELECT を排除 (#31)

- INSERT イベント受信ごとに全件 SELECT していた方式を、payload ベースの差分更新に切り替えた
- 起動時に players・tables をキャッシュし、INSERT ペイロードをキャッシュで手動組み立てすることで試合登録時の SELECT を不要にした
- キャッシュミスや対象 match が未存在の場合は全件再取得にフォールバック。DELETE と名前変更（UPDATE）は差分更新が困難なため引き続き全件再取得

### 設計の方針

Realtime による同期は v1.1.0 で導入したが、パフォーマンス上の問題が3つ積み重なっていた。まず「1登録→5回 SELECT」、次に「他大会のイベントを誤検知」、最後に「INSERT のたびに全件取得」。これらを独立した問題として1つずつ潰した。差分更新はキャッシュとの整合が崩れるリスクを伴うため、フォールバック戦略を明示して Safety Net を確保している。
